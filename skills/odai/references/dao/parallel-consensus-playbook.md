# 并行合议手册

## 何时读取

- 判能否开完整增强、三席模拟或辅助复核
- 判何时升档、何时降档、何时回原流程
- 需冻后独立复查，或需给子 agent 组统一下发包

## 公共定义（后文皆引此）

- **真实可用模型**：当前用户在当前会话、当前环境可选/可分配/可直调者。候选名单、后台库存、旧日志、历史权限、理论可查而未回填者，皆不算。用户自述、模型自述、旧缓存、需额外安装/补录/手工操作方得者，皆不作证。原生派位、宿主/API 代查或实际拉起若回"超出成本层级 / 权限不足 / 额度不足 / 当前账号不可用"或等价拒绝者，即剔除。Copilot 子 agent / 派位链路中，理论成本标签只作记录；惟实际成本拒绝可据剔。
- **同版 odai 契约**：每席至少承同版 canonical 入口、当前命中模块、必读 support files、当前题面/边界/验收/不可擅自假设项；不能可靠下发，则不入任何并行形态。
- **合法阶段**：确认前只许问题理解与提问准备；确认后方可再分析、方案收束、冻后独立复查或执行分工。
- **降档**：「当前模型三席模拟」「子 agent 辅助复核」「原流程」皆降档；不得称"多模型合议"或"完整增强"。

## 四种模式

- **完整增强**：多 agent、多模型，同题并判；至少三种当前真实可用之不同模型。
- **三席模拟复核**：三席同模型、异关注点交叉复核（异构未坐实时）。
- **辅助复核**：一至二席补证、反证或辅助复查。
- **原流程**：不并行；冻后之执行分工亦归此类。

## 模式成立条件

### 完整增强

1. 当前环境明确为 Copilot，且可稳定拉起并行子 agent。
2. 已尽直查、AI 续查与宿主/API 续查，可证至少三种真实可用之不同模型（见公共定义）。
3. 同版 odai 契约可可靠下发。
4. 任务属中/大复杂，并行收益胜额外成本。
5. 当前阶段合法。

### 三席模拟

1. Copilot；至少三席可稳定拉起，同版契约可可靠下发。
2. 已尽自查仍不能合法坐实三种真实可用之不同模型。
3. 三席可可靠沿用主流程当前模型；模型标识未露者，至少可可靠沿用同路默认模型。
4. 任务仍中/大复杂，三席交叉值得其本。
5. 阶段合法。

### 辅助复核

1. Copilot；至少一席子 agent 可稳定拉起。
2. 同版契约可可靠下发。
3. 本轮确有补证、反证或辅助复查之收益。
4. 阶段合法。

异构未坐实者，只称"辅助复核"，不得称"多模型合议"。

## 模型能力自查口径

- 先查当前上下文、工具直出、会话日志/缓存、平台返回、会话资源与宿主同会话回填。
- 本地自查不足，回交总控；总控行 AI 续查；仍不足，则通知宿主/API 续查。
- 同会话、同环境内尚有未试取数路径者，不得以"当前环境未暴露"为终断口径。
- "真实可用"判定见公共定义。

## 真实增强自测与验收

本轮本就在测完整增强，或欲凭新暴露之模型列表首次自动升档时，须补最小真实派位探测：

1. 先定主流程当前模型，并取至少两个已过滤后之不同候选。
2. 运行面支持显式选模者，对同一只读小任务分别显式发起三席最小探测。
3. 三席皆成功且无拒绝，方记"当前会话已验证可用"。
4. 候选虽露而运行面不支持显式选模、或不能稳定将模型 id 传入席位者，最多算"候选已暴露"，不得称"真实增强已验"。
5. 收束须写：本轮实际探测之模型、被剔除者及其因、最终是否达三模型显式成功。

## 启动前先做触发判定

不以"能并行"即必并行。

### 强触发

- 同时存在两条以上合理主路，取舍会改写范围/边界/验收/成本/风险接受度。
- 任务跨两个以上专项模块，模块间有真实牵引。
- 今断属高代价冻结点，如方向定案、规格冻结、架构主路或广范围审查结论。
- 原流程已收束一轮，仍遗会改写路线/边界/验收/不可受结果之剩余分歧。
- 误判回滚与纠偏代价显著高于一次并行复核之本。

### 排除

- 单文件小改、小范围修辞、已知模式下之低风险修补。
- 主路单一、替代稀少、误判纠偏甚廉。
- 当前只是已冻方案下之执行分工、补证、实现或文档整理。

### 增强尝试链

- 命中中/大复杂且未触排除者，依次试：完整增强 → 三席模拟 → 辅助复核 → 原流程。
- 用户纵明言要增强，条件不立亦不得硬凑；只停本轮最高合法档位。
- 原流程推进中若新生主路冲突、跨模块牵引或高代价冻结点，立时重判。
- 问题已压成单主路、低风险或执行分工者，及时回原流程，不为形式留并行。

## 拉起子 agent 时之统一约束

- 主流程与诸席位皆承同版 odai 契约（见公共定义）；不得主流程守规而子 agent 自由发挥。
- 一旦拉起子 agent，先读 `assets/dao/subagent-execution-template.md` 组统一下发包；不得以散乱 prompt 临时拼凑。
- 须席位选模，再读 `references/dao/model-selection-baseline.md`。
- 每个下发包只对一席；不得多席混于一包。
- 不能确认契约已可靠下发者，不得入任何增强模式。

## 主流程专属权限

- 是否开增强、何时降档/回原流程，皆主流程断之。
- 席位数、模型、关注点、是否冻题、是否独立复查或执行分工，皆主流程定之。
- 子 agent 只许在既定题面、边界与关注点内分析/挑战/补证/执行分工；不得自改题面、擅换主路、再拉新 agent，亦不得替主流程升级给用户。

## 统一下发包最小字段

1. canonical 入口路径
2. 当前命中模块路径
3. 必读 support files
4. 当前阶段与本席关注点；执行分工者，写分工范围
5. 本席 agent、模型与选模理由
6. 今判、所求、所重
7. 已知、已验、未定、相左
8. 当前题面、范围 / 非范围、成功标准、不可受结果
9. 禁止擅自假设项、输出要求、失败回退口径
10. 当前已暴露之权限、配额、实际派位拒绝信息及其处理

## 阶段规则

### 确认前

- 只许整理问题定义草案、补证、提反证、校题面、备结构化问题。
- 对外只落：今判、已知、未定与问题组。
- 不得冻方案、定主路、入实施。

### 确认后

主流程先冻最小交付，再行并行：

- 当前问题定义
- 范围、非范围与权限边界
- 成功标准、不可受结果与验收口径
- 当前可用证据与禁止擅自假设项
- 本轮待收敛之核心问题

### 首轮合议

- 完整增强下，至少三席、至少三种不同模型；其中至多一席与主流程同模型。
- 默认至少设：一席反证/逆向挑战、一席风险/边界/验收、一席补证/实施代价/冻后复查。
- 关注点只为补挑战，非拆题分工；每席皆围绕同一核心问题给相对完整之判断。

### 收束与复议

主流程先三事：去重、裁冲突、查遗漏。

- 能内部收束者，不交用户。
- 惟剩余分歧仍足改写路线/边界/验收/不可受结果者，始交用户复核。

### 冻后独立复查

- 输入以冻方案、关键证据与今断为主；不转交整段讨论史。
- 能换模则换模；不能则至少换关注点。
- 新证若足摇冻判，立刻回写上游。

### 执行分工

- 方案既冻，方可拆实现/验证/补证/文档等子任务。
- 此属原流程执行组织，不得称增强。
- 各席仍承同一冻题、主路、边界与验收。

## 输出与留痕

- 收束须写本轮模式、实命中模块、实调用 agent、实调用模型；未暴露者直书"当前环境未暴露"。
- 未入完整增强者，须直陈因由：模型列表未露、仅一模型、异构不足、三席模拟不立、辅助复核亦不值，或实际拒绝所阻。
- 不转储无益争论、不堆原始分歧；只留最终判断、关键反证、是否需回写上游。
- 子 agent 输出越界者，主流程以失约论，回原流程重整。
