# Planning Playbook

## 何时读取

当你需要以下任一信息时，再读取本文件：

- 如何处理附件、截图、原型、流程图或其他材料
- 用户输入很少时，如何先形成一版带前提的方案
- 该怎样分层提问，避免一轮一问
- 工作草案至少该写到什么程度
- 如何主动扩展相邻场景、隐藏约束与候选解释

## 材料与低信息输入

若用户提供了附件材料或非纯文本内容，先做材料分析，再做方案设计。至少分析：

- 材料表达了什么目标或意图
- 有哪些显性规则、隐性假设和遗漏信息
- 与现有功能是否一致
- 与项目现状是否冲突
- 哪些结论可直接采用，哪些必须向用户确认

若用户几乎没给需求，只给了一个参考，或只说“做一个 XX”，甚至连参考都没有，也不能只回“请补充信息”。

此时默认先做：

1. 检查项目级说明和文档。
2. 查找项目内可复用的相似功能、已有模块、已有模式和约束。
3. 主动扩展相邻场景、边界条件、失败路径、上下游依赖和隐藏成本，不要只围着用户原话打转。
4. 若项目内没有足够参照，再基于常见做法给出带前提的默认方案。
5. 先形成初步方案，再把真正会影响方向的关键不确定项一次性问给用户。

若需要自行设计，必须明确标注：

- 哪些是基于研究或经验给出的默认设计
- 哪些地方仍需要用户拍板

## 主动扩展与候选假设

主动扩展不是随便脑补，而是为了提高判断质量，主动把用户没说全、可能说错或尚未意识到的部分提前拉出来。

默认至少主动检查：

- 相邻使用场景和角色变化
- 边界条件、空输入、异常输入和失败路径
- 上下游依赖、数据来源、权限前提和环境差异
- 用户口头目标与真实使用目标是否一致
- 用户声称的原因、方案或限制是否与项目事实冲突
- 后期维护成本、解释成本、排查成本和协作成本

执行要求：

1. 主动扩展出来的内容只能写成候选解释、假设项、风险项或待确认项。
2. 若主动扩展触及方向、边界、验收、风险或成本取舍，就必须问用户确认，或补足可验证证据。
3. 若能从项目、材料、代码、日志或低风险验证中先自证，就优先先自证，不急着追问用户。

## 步进式询问

提问必须分层，但同一层的问题要一次性问完。

默认分四层：

1. 目标层：要解决什么、想达成什么、为什么现在做
2. 场景层：谁来用、何时用、在哪个流程里用、和现有功能怎么衔接
3. 边界层：本次包含什么、不包含什么、有哪些约束和优先级
4. 落地层：交付形式、验收方式、时间预期、兼容要求

执行要求：

1. 先判断当前卡在哪一层。
2. 只问当前最影响判断的那一层，必要时顺带少量下一层高关联问题。
3. 单轮问题尽量控制在 3 到 7 个高价值问题。
4. 若已有信息足够，就不要为凑流程硬问。
5. 若用户回答后产生新分歧，再进入下一轮，而不是提前把所有问题一次性铺满。
6. 提问前先给出当前理解、推荐默认项或候选方向，再让用户确认。
7. 若只差一步拍板就能定稿，优先给推荐项加选项，不要让用户重新组织整段需求。

## 多角度分析

做方案或诊断时，至少综合考虑：

- 用户目标与真实动机
- 用户表述背后的真实需求、真实意愿与真正痛点
- 用户表述里可能说错、说漏或混淆层级的部分
- 业务流程与使用场景
- 现有功能与既有规则
- 实现复杂度与维护成本
- 扩展性与后续演进空间
- 数据、权限、兼容性与风险
- 用户是否真的想清楚了需求
- 若是 bug，现象与根因是否混淆、证据链是否完整、复现条件是否充分

若这些维度互相冲突，要指出冲突点并向用户确认取舍。

## 工作草案默认结构

只有在当前信息足够、且确实需要完整方案时，再按需输出以下部分：

1. 需求理解
2. 目标与非目标
3. 关键约束与风险
4. 方案选项
5. 推荐方案
6. 执行拆解
7. 用户文档
8. AI 文档
9. 待确认项
10. 文档更新记录

若当前只是探索期，优先压缩为：

1. 当前理解
2. 临时判断或候选方案
3. 待确认项
4. 本轮建议下一步

这四部分本身也应视为“第一版工作文档”，不是纯聊天总结。

若存在明显冲突、疑点或多种解释路径，默认还要补一小段“为何暂不采信用户原话 / 当前更可信的候选解释是什么”，避免文档只剩表面复述。

## 支持子 agent 的环境优化

仅当运行环境被明确识别为支持 subagent，且多代理并行能明显提升收敛效率时，才允许并行借力。

默认可拆的侧车方向包括：

- 项目级约束与现有功能梳理
- 用户输入与材料解析
- 方案设计与执行路径
- 风险与冲突复核

规则：

1. 主 agent 先给边界，避免子 agent 擅自扩题。
2. 每个子 agent 都应形成相对完整判断，不只给碎片意见。
3. 允许结论不一致，但必须由主 agent 汇总、去重、裁冲突、查遗漏。
4. 最终只对用户输出一份统一口径文档。
5. 小任务、短上下文、范围清晰时不要滥用 subagent。
