# Workflow Kit

## 适用时机

当你需要以下任一信息时，再读取本文件：

- 项目内默认应该落哪些文档
- 主文件与执行单如何分工
- 任务状态如何流转
- 三种常见场景各该怎么起步

## 默认产物包

若项目没有既有约定，优先采用这套最小产物：

```text
plans/
  2026-04-09-任务名.md
  2026-04-09-任务名.execution.md   # 仅在实施步骤较长时创建
```

这套默认产物属于 harness 常用的最小落盘方式；若任务需要，也可以直接写入项目里其他正式文档位置。

### 主文件职责

主文件默认承载：

1. 背景与目标
2. 工作草案
3. 过堂结论
4. 执行结论
5. 文档更新记录

若进入实施或需要收口，再补：

1. 进度记录
2. 验证与收口

### 执行单职责

只有在以下情况才拆出 `.execution.md`：

- 实施步骤较长，主文件会失焦
- 需要多人协作或多轮续跑
- 需要更清晰地跟踪步骤、验证和回滚

执行单最少包含：

1. 目标
2. 范围与非范围
3. 步骤清单
4. 验证方式
5. 风险与回滚

## 状态速查

```text
draft -> reviewed -> ready -> executing -> verified
   \         \         \
    \         \         -> blocked
     \         -> blocked
      -> blocked
```

状态含义：

- `draft`：方向初步形成，但还有关键待确认项
- `reviewed`：已完成过堂，拿到了裁决
- `ready`：可以直接按下一步或执行单推进
- `executing`：已进入落地执行
- `verified`：已完成验证与收口
- `blocked`：当前被关键信息、权限、环境或分歧卡住

## 三类常见起手式

### 1. 模糊需求

用户输入示例：

```text
想把支付页改一下，但我自己也没完全想清楚。
```

默认动作：

1. 进入收敛入口
2. 采用草案模式
3. 建主文件并先写工作草案
4. 最后只问 1 到 3 个真正影响方向的问题

### 2. 已有方案待审

用户输入示例：

```text
这是我写的方案草稿，帮我看下有没有明显风险。
```

默认动作：

1. 进入复核入口
2. 采用过堂模式
3. 先抽取最小交接载荷
4. 再输出三省六部结论和门下省裁决

### 3. 已授权直接推进

用户输入示例：

```text
方案就按这个来，直接帮我落地。
```

默认动作：

1. 进入推进入口
2. 判断是否满足执行条件
3. 若满足，进入推进模式并补执行单或直接实施
4. 实施后补验证与收口
